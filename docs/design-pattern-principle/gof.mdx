# Gang of Four

## Refactoring Guru
- A good site to learn GoF https://refactoring.guru/design-patterns
- Understand **aggregation or composition** vs **inheritance**
> Aggregation or Composition work almost the same way: one object has a reference to another and delegates it some work, whereas with inheritance, the object itself is able to do that work, inheriting the behavior from its superclass.

### Structural Patterns
<table>
    <tr>    
        <th>Pattern Name</th>
        <th>Structure</th>
        <th>Key Point</th>
    </tr>
    <tr>    
        <td>Adaptor</td>
        <td>
            <ul>
                <li>Client</li>
                <li>Client Interface</li>
                <li>Adaptor</li>
                <li>Service</li>
            </ul>                
        </td>
        <td>
            <ul>
                <li>Convert an interface to suit another different interface</li>
                <li>Used when communicating with third party libraries</li>
            </ul>
        </td>
    </tr>
    <tr>    
        <td>Bridge</td>
        <td>
            <ul>
                <li>Client</li>
                <li>Abstraction</li>
                <li>Implementation</li>
            </ul>
        </td>
        <td>
            <ul>
                <li><a href="https://stackoverflow.com/questions/1425171/difference-between-bridge-pattern-and-adapter-pattern"> Read Stack overflow here</a></li>
                <li>Implementating upfront</li>
                <li>Prevent insane number of classes created due to combinations of several orthogonal hierarchies </li>
                <li>Divide current classes into 2 or more, separate, orthogonal hierarchies</li>
                <li>Decouple abstraction and implementation. Note that the abstraction and implementation here do not mean abstract classes, interface and concrete classes. It is about how we extract the relationship of objects into abstraction and implementation. The abstraction defines what to do. The implementaion focuses on how to do.</li>
            </ul>
        </td>
    </tr>
    <tr>    
        <td>Composite</td>
        <td>
            <ul>
                <li>Client</li>
                <li>Component</li>
                <li>Leaf</li>
                <li>Composite</li>
            </ul>
        </td>
        <td>
            <ul>
                <li>Compose objects into tree-like object structures</li>
            </ul>
        </td>
    </tr>
    <tr>    
        <td>Decorator</td>
        <td>
            <ul>
                <li>Client</li>
                <li>Component</li>
                <li>Decorator</li>
            </ul>                
        </td>
        <td>
            <ul>        
                <li>a.k.a Wrapper, Skin</li>
                <li>1 to 1</li>
                <li>Can chain multiple decorators</li>
                <li>Resulting objects will be structured as a stack</li>
                <li>Contain same set of methods as the target object</li>
                <li>may alter the result by doing something either before or after it passes the request to the target</li>
                <li>e.g. io streams, compress then encrypt</li>
            </ul>
        </td>
    </tr>
    <tr>    
        <td>Facade</td>
        <td>
            <ul>
                <li>Client</li>
                <li>Facade</li>
                <li>Additional Facade</li>
            </ul>
        </td>
        <td>
            <ul>
                <li>Simplified Interface</li>
                <li>Hide complex and unused interface</li>
            </ul>
        </td>
    </tr>    
    <tr>    
        <td>Flyweight</td>
        <td>
            <ul>
                <li>Client</li>
                <li>Context</li>

                <li>Flyweight</li>
            </ul>

                <li>Share common immutable states</li>
                <li>Reduce number of repeated objects</li>
                <li>Save Memory</li>
            </ul>
        </td>
    </tr>
    <tr>

        <td>

            </ul>
        </td>
        <td>
            <ul>
                <li>Exactly the same interface as service interface</li>
                <li>Useful for unit test by swapping the implementation with proxy object</li>
            </ul>
        </td>
    </tr>

        <td>Memento</td>
        <td>
            <ul>
                <li>Originator</li>

                <li>Caretaker</li>            
            </ul>
        </td>
        <td>
            <ul>
                <li>Memento = Snapshot</li>
                <li>Originator is the original object that hold the state</li>
                <li>Originator creates Memento</li>
                <li>Memento links to originator</li>
                <li>Caretaker stores Memento</li>
                <li>Honor security of an object. Caretaker do not access the state of memento.</li>
            </ul>
        </td>
    </tr>
